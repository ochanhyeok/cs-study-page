스레드 풀 + 작업 큐

200개 일꾼이 1000개 일거리를 5번씩 분배 처리
작업 큐 (1000개)
W1 W2 W3 W4 W5 W6 … W999 W1000
→
스레드 풀 (200개)
T1 T2 T3 T4 T5 … T199 T200
한 스레드의 일생 (1000 ÷ 200 = 5회)
T1 W1W201W401W601W801
T2 W2W202W402W602W802
T200 W200W400W600W800W1000
스레드는 풀에 상주하며 작업 끝나면 큐에서 다음 작업을 꺼냄. 작업은 1회용 — 처리되면 사라짐. 스레드 1000개로 늘리면 race가 줄어드는 게 아니라 더 폭발적으로 발생 — race는 자원 부족 문제가 아니라 동시 접근 문제이기 때문.

// 1) 카운터 1로 시작 CountDownLatch startSignal = new CountDownLatch(1); // 2) 1000개 스레드 모두 출발선 대기 executor.submit(() -> { startSignal.await(); // ← 정지 (CPU 0%) account.withdraw(1); }); // 3) Thread.sleep(100) — 출발선 도착 시간 // 4) 동시 출발 startSignal.countDown(); // ← 카운터 0 → 모두 동시 풀림
비유: 운동회 출발선의 "땅!" 신호. 모든 선수가 출발선에 모인 뒤 신호와 동시에 출발해야 누가 빠른지 정확히 비교 가능. Thread.sleep(100)은 신호 보내기 전 모든 선수가 출발선에 도착할 시간.
숫자 1의 의미 — countDown() 호출 횟수
new CountDownLatch(N) 의 N은 신호가 N번 와야 풀린다 는 의미. 카운터가 N으로 시작 → countDown() 호출마다 1씩 감소 → 0 되면 모든 await() 풀림.
우리는 메인이 신호 한 번만 보내면 200스레드가 동시 출발하면 되므로 N=1.
패턴N 값countDown 호출사용처
A. 출발 신호 (우리) new CountDownLatch(1) 메인이 1번 여러 스레드 동시 출발
B. 종료 대기 new CountDownLatch(워커수) 워커가 끝나며 각자 1번 메인이 모든 워커 끝나기 대기
참고: CountDownLatch는 일회용 — 한 번 카운터가 0 되면 재사용 불가. 재사용 가능한 동기화가 필요하면 CyclicBarrier (1주차에는 안 쓰지만 참고).
03

race condition — 1층에서 일어나는 일

3개 스레드가 거의 같은 시점에 withdraw(1) 호출. balance 초기 100원.
초기 balance
100
t=1t=2t=3t=4t=5t=6
T1 read 100 … … write 99 — —
T2 … read 100 … … write 99 —
T3 … … read 100 … … write 99
주목: T1, T2, T3 모두 balance=100을 봄. 셋 다 "100 - 1 = 99" 계산 후 같은 99를 덮어씀.
실제로는 3원 출금되어야 하는데, balance는 99 (1만 차감)로 남음.
race condition 본질
if (balance > 0) { balance--; }는 사실 read-modify-write 3단계. 이 사이의 "틈"에 다른 스레드가 끼어들면 결과가 망가짐. 1000개 시도 시 100명만 성공해야 하는데 120~150명 성공 + 잔고 음수로 떨어짐.
한 줄 코드의 진짜 모습 — read-modify-write 3단계
balance-- = 사실은 3개의 CPU 명령
1
LOAD balance
메모리에서 100 읽기 (read)
2
SUB 1
100 - 1 = 99 계산 (modify)
3
STORE balance
메모리에 99 쓰기 (write)
비원자적 = 1, 2, 3 사이에 다른 스레드가 끼어들 수 있음 → race 발생
원자적 = 1~3이 쪼갤 수 없는 한 덩어리로 일어남 → 끼어들 수 없음 → race 없음
해결책: synchronized (락 + 메모리 배리어로 1~3을 묶음) 또는 AtomicInteger (CAS로 1~3을 한 번에)

05

JIT 최적화 전후

빈 루프는 hot spot. JIT가 변수를 레지스터에 캐싱해버린다.
컴파일 전 (인터프리터)
while (!stopped) { // 빈 루프 — 매 사이클 stopped를 메모리에서 읽음 }
매 사이클마다 메모리 접근. 가시성 위반이 약하게 보이거나 안 보일 수 있음.
↓
JIT 컴파일 후 (native)
if (!stopped) { // 한 번만 체크 while (true) { } // 영원 루프로 고정 }
stopped가 레지스터에 캐싱됨. 다른 스레드의 변경을 영영 보지 못함.
왜 일어나나? JIT(HotSpot JVM)는 자주 호출되는 메서드를 native 코드로 컴파일하면서 "loop invariant code motion" 최적화를 적용한다. "이 루프 안에서 stopped를 안 바꾸네 → 매번 메모리에서 읽지 말고 레지스터에 캐싱하자" 라고 판단. `volatile` 키워드는 JIT에게 "이 변수는 캐싱 금지"를 지시해 이 최적화를 막는다. -Xint 옵션은 JIT 자체를 끄고 인터프리터 모드만 사용 → 가시성 위반의 안전망.

동시성 디버깅의 함정
개발자가 동시성 버그를 의심해 println으로 디버그 → 버그가 사라짐 → "우연이었나" 종료 → production에서 그대로 폭발.
버그를 잡으려고 넣은 도구가 버그를 가린다. 학습 코드 빈 루프(`while (!stopped) {}`)가 정답인 이유 — println, sleep, Logger, BlockingQueue 등 모든 동기화 자료구조가 가시성 문제를 가릴 수 있다.

CAS (Compare And Set) 동작 흐름

AtomicInteger의 정체 — 락 없이 원자성을 보장하는 lock-free 패턴

T2 read 100 check OK CAS 100→99 ✗ retry read 99 check OK CAS 99→98 ✓
read (값 읽기)

CAS 실패 (다른 스레드가 먼저)
retry (재시도)
// AtomicInteger.compareAndSet 흐름 while (true) { int current = balance.get(); // (1) 현재값 읽기 if (current < amount) return false; // (2) 잔고 부족 체크 if (balance.compareAndSet(current, current - amount)) { return true; // (3a) 성공 } // (3b) 실패: 다른 스레드가 그 사이에 바꿈 → (1)부터 재시도 }
CAS의 본질: "지금 값이 current면 new로 바꿔줘" 를 CPU 단일 명령어로 처리.
  • x86 / x64: CMPXCHG
  • ARM (Apple Silicon): LDREX / STREX
어느 단계에서도 락은 없음 — lock-free. 충돌 시 재시도 패턴이라 임계 영역이 짧을수록 효율적.
09

synchronized vs CAS — 두 가지 접근 비교

비관적 락 vs 낙관적 lock-free — 같은 문제를 푸는 정반대 철학
비관적
synchronized — 락 + 줄 서기
  1. 락 시도 → 획득 (다른 스레드 대기)
  2. 임계 영역 실행 (read + write)
  3. 락 해제 → 다음 스레드 차례
🚪 화장실 비유: 들어가서 문 잠금. 199명은 줄 서서 대기. 한 명 끝나야 다음 한 명.
낙관적
AtomicInteger / CAS — 락 없이 시도
  1. 현재값 읽기 (락 X)
  2. compareAndSet 시도 (원자적)
  3. 성공 → 끝 / 실패 → 재시도
🚪 화장실 비유: 일단 들어감. 나오기 직전 "옆에 누가 같이 있나?" 확인. 있으면 다시 시도. 줄 안 섬.
측면 synchronized AtomicInteger (CAS)
락 사용 ✅ 잡음 ❌ 안 잡음 (lock-free)
충돌 시 동작 줄 서서 대기 재시도 (busy retry)
컨텍스트 스위칭 자주 발생 거의 없음
적합한 상황 긴 임계 영역, 복잡한 로직 단순 카운터/플래그, 짧은 연산
불리한 상황 짧은 작업 (오버헤드 비중 ↑) 충돌 매우 잦음 (재시도 폭증)
현대 JVM (Java 21)에서: biased locking, lightweight locking, adaptive spinning 등 최적화 덕에 단순 임계 영역에서는 두 도구의 차이가 11% 안팎 — 1990년대 자료의 "synchronized 30배 느림"은 옛날 이야기. 실용적 선택 기준은 성능보다 가독성과 의도 표현인 경우가 많음.

race vs visibility — 1주차 두 큰 줄기

이름은 비슷하지만 메커니즘이 완전히 다른 두 동시성 문제
결과가 깨짐
race condition (atomicity 위반)
  1. 한 변수에 read-modify-write
  2. 두 스레드가 거의 동시에 read
  3. 같은 값으로 write → 덮어쓰기
  4. 결과: 누락 / 초과 / 음수
💥 예: 100원짜리 통장에서 200스레드가 동시 출금 → 잔고가 -5원으로 떨어짐
통신이 끊김
visibility 위반
  1. 한 스레드가 변수 변경
  2. 변경이 자기 CPU 캐시에만 반영
  3. 다른 스레드는 옛날 캐시 값을 봄
  4. 결과: 신호 못 받음 / 무한 루프
💥 예: 메인이 stopped=true 했는데 Worker는 5초 후에도 false로 보고 무한 루프
측면 race condition visibility
무엇이 깨지나 결과 (값 자체) 통신 (변경 전달)
발생 위치 한 변수의 read-modify-write 사이 한 스레드의 변경이 다른 스레드에
근본 원인 동시 접근으로 인한 인터리빙 CPU 캐시 격리 + JIT 최적화
시연 도구 ExecutorService + CountDownLatch 빈 루프 Worker 스레드
해결 도구 synchronized, AtomicInteger volatile, synchronized
1주차 STAGE STAGE 1, STAGE 3 STAGE 2
둘은 별개 문제:
  • synchronized — 락(원자성) + 메모리 배리어(가시성) → 둘 다 풀어줌
  • AtomicInteger — CAS(원자성) + 내부적으로 volatile(가시성) → 둘 다 풀어줌 (단순 연산만)
  • volatile — 메모리 배리어만 → 가시성만 풀어줌 (count++ 같은 read-modify-write 못 풀어줌)
그래서 카운터에는 volatile 만 붙이면 부족하고, 깃발(boolean stopped)에는 volatile 로 충분.
